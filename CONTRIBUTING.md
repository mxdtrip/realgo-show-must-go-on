# Участие в разработке

Этот документ описывает правила работы с ветками, коммитами и pull request в монорепозитории FreeBurger.

## Основные принципы

- Не вносите изменения напрямую в `main`. Работайте через отдельную ветку и pull request.
- Один pull request должен решать одну задачу или одну связанную группу изменений.
- Не смешивайте функциональные изменения с несвязанным рефакторингом или форматированием.
- Не добавляйте секреты, токены, приватные ключи и локальные `.env`-файлы в Git.
- Перед отправкой изменений обновите документацию, тесты и примеры окружения, если изменилось поведение или конфигурация.

## Имена веток

Используйте короткое имя в нижнем регистре с дефисами:

```text
<type>/<scope>-<description>
```

Примеры:

```text
feat/web-auth-page
fix/extension-popup-crash
refactor/api-user-service
docs/repo-contributing-rules
```

Допустимые `type` совпадают с типами коммитов ниже. Если работа связана с задачей, её номер можно добавить в начало описания: `feat/web-123-auth-page`.

## Коммиты

Сообщения коммитов оформляются по спецификации Conventional Commits:

```text
<type>(<scope>): <description>
```

Примеры:

```text
feat(web): add sign-in form
fix(extension): restore popup state
refactor(api): extract token validator
docs(repo): document local setup
```

### Типы коммитов

- `feat` — новая пользовательская функциональность.
- `fix` — исправление ошибки.
- `refactor` — изменение внутренней реализации без изменения поведения.
- `perf` — улучшение производительности.
- `test` — добавление или изменение тестов.
- `docs` — изменения документации.
- `style` — форматирование без изменения логики.
- `build` — зависимости и система сборки.
- `ci` — CI/CD и автоматизация.
- `chore` — техническое обслуживание репозитория.
- `revert` — отмена предыдущего изменения.

### Scope монорепозитория

- `web` — `apps/web`.
- `extension` — `apps/extension`.
- `api` — `services/api`.
- `ui` — `packages/ui`.
- `shared` — `packages/shared`.
- `config` — `packages/config`.
- `repo` — корневые настройки и документация репозитория.

Scope можно опустить, если изменение действительно охватывает весь репозиторий. Не используйте имя файла или номер задачи вместо scope.

### Требования к сообщению

- Пишите заголовок на английском языке, в повелительной форме и без точки в конце.
- Делайте заголовок коротким и конкретным; рекомендуемая длина — до 72 символов.
- Один коммит должен содержать одно логически завершённое изменение.
- Не используйте бессодержательные сообщения вроде `fix`, `changes`, `update` или `work`.
- Для сложного изменения добавьте тело коммита с объяснением причин и принятых решений.

Несовместимое изменение отмечается `!` и поясняется в footer:

```text
feat(api)!: change authentication response

BREAKING CHANGE: clients must read the token from the data field.
```

## Pull request

### Перед созданием PR

- Обновите ветку относительно актуального `main` и устраните конфликты.
- Запустите форматирование, lint, typecheck, тесты и сборку для затронутых приложений или сервисов.
- Проверьте, что в diff нет секретов, временных файлов, сгенерированных артефактов и отладочного кода.
- Добавьте или обновите тесты для изменённого поведения.
- Проверьте изменения вручную в тех средах, которых они касаются: web, браузерное расширение или API.

### Оформление PR

- Заголовок PR оформляйте как Conventional Commit, например `feat(web): add sign-in form`.
- В описании укажите, что изменено, зачем это нужно и как результат был проверен.
- Привяжите задачу или issue, если она существует.
- Для визуальных изменений приложите скриншоты или короткую запись экрана.
- Отдельно отметьте миграции, новые переменные окружения, изменение разрешений расширения и breaking changes.
- Не запрашивайте review, пока обязательные проверки CI не завершились успешно.

Draft PR можно создавать для раннего обсуждения. Перед переводом в Ready for review автор должен самостоятельно пройти checklist из шаблона PR.

### Review и merge

- Все замечания review должны быть исправлены или явно обсуждены до merge.
- После существенных изменений запросите повторное review.
- Не закрывайте чужие обсуждения без ответа автора замечания.
- Предпочтительный способ объединения — squash merge, чтобы заголовок PR стал итоговым Conventional Commit в `main`.
- Удалите рабочую ветку после успешного merge.
